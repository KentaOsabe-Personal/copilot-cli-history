# 要件定義書

## プロジェクト記述（入力）
**課題:** GitHub Copilot CLI のローカル会話履歴を読み返す利用者は、セッション一覧でどのプロジェクトに対して実行したセッションなのかを素早く判断しにくい。
実行ディレクトリの一部を覚えていても、現状の一覧検索ではその情報を使って絞り込めない。

**現状:** アプリは current 形式の `workspace.yaml` から実行ディレクトリを読み取る実装と、保存済み read model に `cwd` / `git_root` を持つ schema を備えている。
一方、実運用 DB ではそれらのカラムが全て `null` になっており、一覧 UI でも実行ディレクトリは明示表示されていない。検索は保存済み `search_text` を使うが、実行ディレクトリは検索対象外である。

**変更したいこと:** current 形式セッションの実行ディレクトリを保存済み read model の正規化メタデータとして保持し、一覧カードで表示し、一覧検索で絞り込めるようにする。
検索のためだけに `search_text` へ情報を閉じ込めるのではなく、表示と検索の根拠になる実行ディレクトリ情報を read model 上で利用可能にする。

## はじめに
この仕様は、GitHub Copilot CLI のローカル会話履歴を読み返す利用者が、セッション一覧で実行ディレクトリを確認し、プロジェクトパスやディレクトリ名を使って目的のセッションへ辿り着けるようにするための要件を定義する。

対象は、実行ディレクトリを保存済みセッションのメタデータとして利用可能にすること、一覧カードに実行ディレクトリを表示すること、一覧検索で実行ディレクトリに一致するセッションを絞り込めることに限定する。

## 境界コンテキスト
- **対象範囲**: current 形式セッションの実行ディレクトリメタデータ、保存済み read model 上の `cwd` / 必要に応じた `git_root` の利用可能性、一覧カードでの実行ディレクトリ表示、一覧検索対象への実行ディレクトリ追加、検索フォーム説明、同期後の再検索可能状態、backend / frontend tests
- **対象外**: raw files に存在しない legacy セッションの実行ディレクトリ推測、repository / branch / model 専用フィルタ、検索結果スコアリング、検索語ハイライト、semantic search、外部検索サービス、詳細画面内検索、履歴編集・削除・共有、自動同期、認証・認可
- **隣接前提**: 既存の明示同期、保存済み read model、日付範囲、全文検索、一覧 API response shape、read-only 閲覧体験を維持する。既存 row は明示同期により再生成可能である。

## 要件

### 要件1: 実行ディレクトリをセッションメタデータとして利用可能にする
**目的:** As a ローカル会話履歴を読み返す利用者, I want セッションの実行ディレクトリが保存済みメタデータとして扱われてほしい, so that 一覧表示と検索が同じ根拠でプロジェクトを識別できる

#### 受け入れ基準
1. When current 形式のセッションが実行ディレクトリ情報を含む状態で同期される, the Copilot History Application shall その実行ディレクトリをセッション単位のメタデータとして利用可能にする。
2. When 実行ディレクトリ情報がセッション単位のメタデータとして利用可能になる, the Copilot History Application shall 一覧表示と一覧検索の両方から同じ実行ディレクトリ情報を参照できる状態にする。
3. If current 形式のセッションに実行ディレクトリ情報が存在しない, the Copilot History Application shall 推測値または不明 placeholder を生成しない。
4. While 保存済みセッションが明示同期によって再生成される, the Copilot History Application shall 実行ディレクトリメタデータを失わせずに更新する。
5. The Copilot History Application shall 実行ディレクトリ情報を検索用テキストだけに閉じ込めず、セッションの作業メタデータとして利用可能にする。

### 要件2: 一覧カードで実行ディレクトリを確認できる
**目的:** As a セッション一覧を確認する利用者, I want 各セッションの実行ディレクトリを一覧上で読みたい, so that どのプロジェクトに対する会話か詳細を開く前に判断できる

#### 受け入れ基準
1. When 利用者が実行ディレクトリ情報を持つセッション一覧を表示する, the セッション履歴 UI shall 各セッションカードに実行ディレクトリを表示する。
2. When セッションが repository または branch 情報を持つ, the セッション履歴 UI shall repository または branch の表示によって実行ディレクトリ表示を隠さない。
3. If セッションが実行ディレクトリ情報を持たない, the セッション履歴 UI shall 実行ディレクトリの空項目、不明値、または推測値を表示しない。
4. While 実行ディレクトリが長いパスである, the セッション履歴 UI shall 一覧カードの横幅を破綻させずに利用者が値を読める状態を保つ。
5. The セッション履歴 UI shall 既存の会話 preview、表示日時、例外シグナル、詳細導線を維持したまま実行ディレクトリを表示する。

### 要件3: 実行ディレクトリで一覧を検索できる
**目的:** As a 過去セッションを探す利用者, I want 実行ディレクトリの一部文字列で一覧を絞り込みたい, so that プロジェクト名やパスから目的のセッションを見つけられる

#### 受け入れ基準
1. When クライアントが実行ディレクトリに含まれる検索語を指定してセッション一覧を要求する, the Backend Session API shall その実行ディレクトリに一致するセッションを一覧対象に含める。
2. When クライアントが検索語と日付範囲を指定してセッション一覧を要求する, the Backend Session API shall 実行ディレクトリ検索と既存の日付範囲条件の両方に一致するセッションだけを一覧対象にする。
3. When クライアントが会話本文、会話 preview、または issue 由来の検索語を指定する, the Backend Session API shall 既存の検索対象による絞り込みを維持する。
4. If 実行ディレクトリ検索に一致するセッションが存在しない, the Backend Session API shall 失敗応答ではなく空の検索結果として返す。
5. The Backend Session API shall 実行ディレクトリ検索を追加しても既存の一覧 response shape、件数 meta、degraded 状態、issue 情報を維持する。

### 要件4: 検索対象の範囲を実行ディレクトリに限定して拡張する
**目的:** As a プロダクト保守者, I want 今回の検索拡張範囲が明確であってほしい, so that repository や model などの後続フィルタを別仕様として扱える

#### 受け入れ基準
1. Where 一覧検索が保存済みセッションメタデータを対象にする, the Copilot History Application shall 今回の追加対象を実行ディレクトリに限定する。
2. The Copilot History Application shall この feature によって repository、branch、selected model の専用フィルタを追加しない。
3. The Copilot History Application shall この feature によって検索結果スコアリング、検索語ハイライト、semantic search、または外部検索サービスを必須にしない。
4. The Copilot History Application shall raw files に実行ディレクトリが存在しないセッションへ実行ディレクトリを推測付与しない。
5. The Copilot History Application shall 通常表示と検索表示のどちらでも保存済み read model を参照し、一覧検索時に raw files を直接検索しない。

### 要件5: 検索 UI が実行ディレクトリ検索を利用者に示す
**目的:** As a 一覧検索を使う利用者, I want 実行ディレクトリも検索対象だと分かる, so that プロジェクトパスを使って自然に検索できる

#### 受け入れ基準
1. When 利用者が一覧画面の検索フォームを見る, the セッション履歴 UI shall 実行ディレクトリが検索対象に含まれることを示す。
2. When 利用者が実行ディレクトリ由来の検索語を適用する, the セッション履歴 UI shall 現在の日付範囲を維持したまま検索結果を表示する。
3. When 利用者が検索語を解除する, the セッション履歴 UI shall 現在の日付範囲を維持したまま検索語なしの一覧を表示する。
4. While 実行ディレクトリ検索が適用されている, the セッション履歴 UI shall 現在の検索語と日付範囲を利用者が確認できる状態を保つ。
5. If 検索条件が受け付けられない値として指定された, the セッション履歴 UI shall 一覧取得失敗と区別できる形で検索条件の見直しを促す。
